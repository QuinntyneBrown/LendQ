# Module 19: Recurring Loans

**Requirements**: L1-15

## Overview

The recurring loans module allows creditors to set up loan templates that automatically generate new loan instances on a defined schedule. Each recurrence creates a new loan with the same terms, borrower, and repayment structure using the existing loan management infrastructure. Creditors can pause, resume, or cancel the recurrence. A Celery worker processes due recurring loans and delegates loan creation to the loan service.

## Public Endpoints

| Method | Path | Description | Auth |
|---|---|---|---|
| `GET` | `/api/v1/loans/recurring` | List recurring loans for the current creditor | Creditor |
| `POST` | `/api/v1/loans/recurring` | Create a new recurring loan template | Creditor |
| `GET` | `/api/v1/loans/recurring/{recurringId}` | Get recurring loan detail including generated loan history | Creditor |
| `PATCH` | `/api/v1/loans/recurring/{recurringId}` | Update recurring loan template fields | Creditor owner |
| `POST` | `/api/v1/loans/recurring/{recurringId}/pause` | Pause recurring loan generation | Creditor owner |
| `POST` | `/api/v1/loans/recurring/{recurringId}/resume` | Resume a paused recurring loan | Creditor owner |
| `POST` | `/api/v1/loans/recurring/{recurringId}/cancel` | Permanently cancel recurring loan generation | Creditor owner |
| `GET` | `/api/v1/loans/recurring/{recurringId}/generated` | List all loans generated by this recurring loan | Creditor |

## Data Model

| Entity | Purpose |
|---|---|
| `recurring_loans` | Template definition with borrower, terms, recurrence interval, status, and generation tracking |
| `generated_loan_records` | Mapping from recurring loan to each generated loan instance with sequence number and timestamp |

Generated loans are standard `loans` rows managed by the existing loan module (Module 3). The `generated_loan_records` table links them back to their recurring loan source.

## Template Rules

Each recurring loan template includes:

- `borrower_id`
- `description_template`
- `principal_amount` and `currency`
- `interest_rate_percent` (optional)
- `repayment_frequency` (weekly, bi-weekly, monthly)
- `installment_count`
- `recurrence_interval` (how often a new loan is created)
- `start_date`
- `end_date` or `max_occurrences` (optional, at least one required if not indefinite)

## Generation Rules

1. A Celery beat task scans for recurring loans where `next_generation_date <= today` and `status = ACTIVE`.
2. For each due recurring loan, the worker creates a new loan via `LoanService` using the template fields.
3. The generated loan receives its own terms version and payment schedule through the standard loan creation flow.
4. A `generated_loan_record` is inserted linking the new loan back to the recurring loan with a sequence number.
5. `total_generated` is incremented and `next_generation_date` is advanced by the recurrence interval.
6. If `max_occurrences` is set and `total_generated >= max_occurrences`, or if `end_date` is set and the next date exceeds it, the recurring loan transitions to `COMPLETED`.
7. Generation is idempotent per recurring loan per date to handle worker retries safely.

## Lifecycle Rules

1. **Active** — new loans are generated on schedule.
2. **Paused** — no new loans are generated; `next_generation_date` is cleared. Existing generated loans are unaffected.
3. **Resumed** — `next_generation_date` is recalculated from the current date and recurrence interval.
4. **Cancelled** — permanent. No future loans will be generated. Existing generated loans are unaffected. This action is irreversible.

## Concurrency

- Recurring loan generation acquires a row-level lock on the recurring loan row to prevent duplicate generation by concurrent workers.
- Pause, resume, and cancel operations are idempotent and reject duplicate terminal transitions.
- Edit operations require `expected_version` to prevent stale writes, returning `409 conflict` on mismatch.
