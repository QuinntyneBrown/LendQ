# LendQ — L2 Detailed Requirements

---

## L1-1: User Authentication & Secure Session Management

### L2-1.1: Login Screen
The system shall display a login screen with email and password fields, a "Sign In" button, a "Forgot Password?" link, and a "Sign Up" link.
**Acceptance Criteria:**
- Login form is centered on all viewports (mobile, tablet, desktop)
- App logo/name "LendQ" is prominently displayed above the form
- Email and password input fields are clearly labeled with placeholder text
- "Sign In" button is full-width within the form and visually prominent
- "Forgot Password?" and "Sign Up" links are visible below the form
- Form includes input validation, loading, disabled, error, and focus states

### L2-1.2: Sign-Up Screen
The system shall display a registration screen with fields for full name, email, password, and confirm password.
**Acceptance Criteria:**
- Sign-up form has fields: Full Name, Email, Password, Confirm Password
- "Create Account" primary button is full-width
- Link back to "Sign In" is visible
- Layout is consistent with the login screen styling
- Success state clearly indicates whether email verification is required before first use
- Responsive across all three breakpoints

### L2-1.3: Forgot Password Screen
The system shall display a forgot-password screen with an email field and a "Send Reset Link" button.
**Acceptance Criteria:**
- Single email input with "Send Reset Link" button
- Success confirmation message/state is designed
- Back to login link is present
- Error, loading, disabled, and success states are defined
- Responsive across all three breakpoints

### L2-1.4: Session Security & Revocation
The system shall manage sessions securely across login, refresh, logout, password reset, user deactivation, and role changes.
**Acceptance Criteria:**
- Short-lived access sessions and renewable refresh sessions are supported
- Logout invalidates the current session and prevents reuse of the current refresh session
- Password reset invalidates all active sessions for the account
- User deactivation or privileged role removal blocks further access without waiting for natural session expiry
- Expired or revoked sessions are handled gracefully in the UI with redirect or re-authentication prompt

### L2-1.6: Sign Out Option
The system shall provide a clearly visible sign-out option accessible from all authenticated screens across all form factors (mobile, tablet, desktop).
**Acceptance Criteria:**
- Desktop: sign-out action is placed at the bottom of the sidebar, visually separated from main navigation, with user identity context (avatar and name)
- Tablet: sign-out action is accessible via a user menu dropdown triggered by the avatar in the header
- Mobile: sign-out action is accessible via a user menu dropdown triggered by the avatar in the header
- Sign-out uses a distinct icon (e.g., log-out) and label "Sign Out"
- Sign-out option is visually consistent with the existing navigation component style
- User menu dropdown (tablet/mobile) also shows the user's name and email for identity confirmation before sign-out

### L2-1.5: Account Protection
The system shall protect authentication flows from brute-force, enumeration, and automated abuse.
**Acceptance Criteria:**
- Login, sign-up, and password-reset flows are rate-limited per IP and per account/email
- Authentication error messages do not reveal whether an account exists
- Password policy is defined and validated on both client and server
- Email verification state is enforced consistently where required
- Suspicious authentication events are audit logged

## L1-2: User Management & RBAC

### L2-2.1: User List Screen
The system shall display a table/list of all users with columns for name, email, role(s), status, and actions (edit/delete).
**Acceptance Criteria:**
- Desktop: full data table with sortable columns (Name, Email, Roles, Status, Actions)
- Tablet: condensed table or card layout with key info visible
- Mobile: card-based list with expandable details
- "Add User" button is visible at the top
- Search/filter bar is present
- Pagination or infinite scroll is indicated

### L2-2.2: Add/Edit User Dialog
The system shall provide a modal dialog for creating or editing a user, with fields for name, email, role assignment (multi-select: Admin, Creditor, Borrower), and status toggle.
**Acceptance Criteria:**
- Modal overlay with form fields: Name, Email, Role(s) multi-select, Active/Inactive toggle
- "Save" and "Cancel" buttons in the dialog footer
- Dialog is responsive (full-screen on mobile, centered modal on tablet/desktop)
- Validation error states shown inline
- Save flow clearly communicates success, failure, and in-flight status

### L2-2.3: Role Management Screen
The system shall display a screen for viewing and managing roles and their associated permissions from a controlled permission catalog.
**Acceptance Criteria:**
- List of roles (Admin, Creditor, Borrower) with descriptions
- Each role shows its permissions as tags/chips
- Only defined permissions from the approved permission catalog can be assigned
- Admin can edit role permissions
- Permission changes require confirmation and are audit logged
- Responsive layout across all breakpoints

### L2-2.4: Delete User Confirmation Dialog
The system shall display a confirmation dialog when deactivating a user.
**Acceptance Criteria:**
- Warning icon and clear message identify the affected user
- Destructive action is visually distinct from cancel action
- Dialog is centered and responsive
- Dialog explains that access is revoked while historical records are preserved

### L2-2.5: Access Revocation & Administrative Audit Trail
The system shall immediately enforce user, role, and permission changes and preserve an audit trail for administrative actions.
**Acceptance Criteria:**
- Deactivating a user prevents new access immediately
- Active sessions are invalidated after user deactivation, password reset, or privileged role removal
- Audit records include actor, target, action, timestamp, before value, and after value where applicable
- Administrative actions are searchable by user and date range

## L1-3: Loan Management & Terms Governance

### L2-3.1: Loans List Screen (Creditor View)
The system shall display all loans created by the creditor in a list/table with borrower name, principal, outstanding balance, next payment due, status, and actions.
**Acceptance Criteria:**
- Desktop: data table with columns - Borrower, Description, Principal, Balance, Next Due, Status, Actions
- Tablet: condensed table with horizontal scroll or card layout
- Mobile: card-based list showing key loan info with tap-to-expand
- "Create New Loan" button prominently placed
- Status badges (Active, Paused, Paid Off, Overdue) with distinct colors
- Filter/search capability

### L2-3.2: Loans List Screen (Borrower View)
The system shall display all loans where the user is the borrower, showing creditor name, principal (read-only), outstanding balance, next payment, and status.
**Acceptance Criteria:**
- Same layout patterns as creditor view but with creditor name column
- Principal and creditor-controlled terms are visually marked as non-editable
- Borrower action options include View Details, Make Payment, and Request Allowed Schedule Changes
- Responsive across all breakpoints

### L2-3.3: Create/Edit Loan Form
The system shall provide a form for creditors to create or edit a loan with fields: borrower, description, principal amount, currency, interest rate, repayment frequency, installment count or maturity date, start date, and notes.
**Acceptance Criteria:**
- Full-page or large modal form with clearly labeled sections
- Borrower dropdown/search select from registered users
- Currency field is fixed by system policy or explicitly selectable
- Principal input uses currency formatting
- Repayment frequency dropdown supports Weekly, Bi-weekly, Monthly, and Custom
- Either installment count or maturity date is required
- Custom schedule option reveals a schedule builder with installment rows
- Generated schedule preview and balance summary are shown before save
- "Save" and "Cancel" buttons are present
- Responsive: single-column on mobile, two-column on desktop

### L2-3.4: Loan Detail Screen
The system shall display complete loan information, payment history, schedule, terms history, and contextual actions.
**Acceptance Criteria:**
- Header section shows loan description, borrower/creditor names, and status badge
- Summary cards show Principal, Total Paid, Outstanding Balance, and Next Payment Due
- Payment history table/timeline shows date, amount, status, and notes
- Upcoming payments schedule section is visible
- Terms history or change history is available
- Action buttons are contextual to role and permissions
- Responsive across all breakpoints

### L2-3.5: Borrower Edit Rules & Loan Change Governance
The system shall enforce field-level edit permissions and approval workflow for creditor-controlled loan terms.
**Acceptance Criteria:**
- Borrowers can view all loan terms but cannot directly edit principal, interest rate, currency, creditor identity, repayment frequency, or installment structure
- Borrower-initiated changes to creditor-controlled terms are submitted as requests with a reason
- Creditors can approve or reject change requests
- Approved changes create a new terms version or equivalent auditable record
- Change history shows actor, old value, new value, timestamp, reason, and approval outcome

## L1-4: Payment Tracking, Scheduling & Balance Management

### L2-4.1: Payment Schedule View
The system shall display the full payment schedule for a loan as a timeline or table showing each payment date, amount, and status.
**Acceptance Criteria:**
- Table/list shows Date, Amount Due, Amount Paid or Remaining, and Status
- Rescheduled payments show original date with strikethrough and new date
- Paused payments are visually distinct
- Partial payments are visually distinct from fully paid installments
- Responsive across all breakpoints

### L2-4.2: Reschedule Payment Dialog
The system shall provide a dialog for eligible users to reschedule a specific payment to a new date.
**Acceptance Criteria:**
- Modal shows current payment date and amount
- Date picker for new date
- Optional reason/note text field
- Impact preview shows how the change affects future schedule dates, maturity, or balance where applicable
- Confirmation state after rescheduling
- Responsive dialog

### L2-4.3: Pause Payment Dialog
The system shall provide a dialog to pause one or more upcoming payments.
**Acceptance Criteria:**
- Modal shows which payment(s) will be paused
- Option exists to pause a single payment, multiple selected payments, or an eligible date range
- Optional reason/note field is present
- Visual indicator shows what the pause affects on the schedule
- Impact preview shows resulting schedule or maturity change before confirmation
- Responsive dialog

### L2-4.4: Record Payment / Lump Sum Dialog
The system shall provide a dialog to record a payment made, including partial and lump-sum amounts that differ from the scheduled amount.
**Acceptance Criteria:**
- Modal includes Amount, Date, Payment Method, and Notes fields
- Scheduled amount is shown and editable
- Clear indication appears when amount differs from scheduled amount
- Running balance preview shows how the payment affects outstanding balance
- Allocation preview shows how the payment is applied when more than one installment is affected
- Submit action shows in-flight state and prevents accidental duplicate submission
- Responsive dialog

### L2-4.5: Payment History with Change Log
The system shall display a history of all payments and payment-related modifications with timestamps.
**Acceptance Criteria:**
- Timeline or table shows payment transactions, reversals, reschedules, pauses, and adjustments chronologically
- Change entries show what changed, old value, new value, who changed it, when, and why
- Visual distinction exists between payments, reversals, and schedule modifications
- Filterable by type (Payment, Reversal, Reschedule, Pause, Adjustment)
- Responsive across all breakpoints

### L2-4.6: Immutable Payment Transactions & Allocation
The system shall record each submitted payment as an immutable transaction allocated to one or more scheduled installments.
**Acceptance Criteria:**
- Each payment has a unique transaction identifier
- Duplicate client retries do not create duplicate payment transactions
- Partial payments and lump-sum payments show allocation across installments
- Reversals or corrections create compensating records rather than destructive edits
- Outstanding balance is derived consistently from authoritative transaction and allocation data

### L2-4.7: Schedule Adjustment Rules & Auditability
The system shall clearly define and display how reschedules and pauses affect future obligations.
**Acceptance Criteria:**
- The system specifies whether downstream due dates shift, maturity extends, or balance/interest changes
- Original and updated schedules are preserved for audit
- Actor, timestamp, and reason are recorded for each schedule adjustment
- Users can preview impact before confirming
- Updated schedule is reflected consistently on loan detail, dashboard, and notifications

## L1-5: Dashboard & Overview

### L2-5.1: Dashboard - Summary Cards
The system shall display summary cards at the top of the dashboard showing key metrics.
**Acceptance Criteria:**
- Cards show Total Lent Out, Total Owed, Upcoming Payments (next 7 days), and Overdue Payments
- Each card has an icon, label, and appropriately formatted value
- Desktop: 4 cards in a row; Tablet: 2x2 grid; Mobile: stacked vertically
- Overdue or warning states are visually distinct

### L2-5.2: Dashboard - Active Loans Table
The system shall display a combined view of active loans on the dashboard.
**Acceptance Criteria:**
- Tabbed or segmented view includes "Loans I Gave" and "Loans I Owe"
- Each tab shows person name, amount, next due, and status
- Quick actions include View and Record Payment where permitted
- Responsive across all breakpoints

### L2-5.3: Dashboard - Recent Activity Feed
The system shall display recent activity as a feed/timeline.
**Acceptance Criteria:**
- Chronological list of recent events with icons, descriptions, and timestamps
- Shows last 10-20 items with "View All" link
- Responsive across all breakpoints

### L2-5.4: Dashboard Freshness & Failure Handling
The system shall present dashboard data in a way that remains usable under partial failure or delayed refresh.
**Acceptance Criteria:**
- Summary, loans, and activity sections can show loading independently
- Error in one section does not block the entire dashboard
- Retry action is available for failed sections
- Data freshness or refresh behavior is clearly defined to the user

## L1-6: Notifications & Alerts

### L2-6.1: Notification Bell & Dropdown
The system shall display a notification bell icon in the top navigation with unread count badge and a dropdown panel listing recent notifications.
**Acceptance Criteria:**
- Bell icon in the header/navbar with numeric badge for unread count
- Clicking opens a dropdown panel with notification list
- Each notification shows icon, message, timestamp, and read/unread state
- "Mark all as read" and "View all notifications" links are present
- Responsive: dropdown on desktop/tablet, full-screen panel on mobile

### L2-6.2: Notification Types & Toast Messages
The system shall display toast notifications for real-time events such as payment received, payment overdue, and schedule changed.
**Acceptance Criteria:**
- Toast appears in top-right corner on larger screens and bottom-center on mobile
- Types include Success, Warning, Error, and Info
- Auto-dismiss after 5 seconds with manual close button
- Toasts stack cleanly if multiple events appear
- Active user sessions receive event-driven notification updates within the defined near-real-time target

### L2-6.3: Full Notifications Screen
The system shall provide a full-page notifications list with filtering.
**Acceptance Criteria:**
- Full list of all notifications with date grouping (Today, Yesterday, Earlier)
- Filter by type: Payments, Overdue, Schedule Changes, System
- Mark individual or all notifications as read
- Responsive across all breakpoints

### L2-6.4: Notification Reliability & Delivery Status
The system shall create and deliver notifications reliably across in-app and email channels.
**Acceptance Criteria:**
- Notifications are deduplicated per event, user, and channel combination
- Delivery attempts have statuses such as Pending, Sent, and Failed
- Failed deliveries are retryable
- Unread count and notification list remain consistent across pages and refreshes
- Transient delivery failures do not silently lose notifications

### L2-6.5: Notification Preferences
The system shall allow users to manage optional email delivery preferences by notification type.
**Acceptance Criteria:**
- User can enable or disable email delivery for supported notification categories
- In-app notifications remain available even when email is disabled
- Preference changes take effect immediately for new notifications
- Preference management is responsive across all breakpoints

## L1-7: Responsive Design

### L2-7.1: Responsive Navigation
The system shall provide adaptive navigation: sidebar on desktop, collapsible sidebar on tablet, bottom tab bar on mobile.
**Acceptance Criteria:**
- Desktop (1280px+): fixed left sidebar with nav items and user avatar
- Tablet (768px): collapsible sidebar with hamburger toggle
- Mobile (375px): bottom tab bar with 4-5 key nav items and overflow menu
- Navigation items reflect current route and role permissions
- Smooth transitions between states

### L2-7.2: Responsive Button & Form Components
All buttons, inputs, selects, and form elements shall be designed for touch and mouse interaction.
**Acceptance Criteria:**
- Buttons have minimum 44px touch target on mobile
- Primary, secondary, destructive, and ghost variants are defined
- Inputs provide labels, focus states, error states, disabled states, and helper text support
- All form components are designed at each breakpoint
- Consistent spacing and typography scale are defined

### L2-7.3: Responsive Dialog & Modal Components
All dialogs and modals shall adapt to viewport size.
**Acceptance Criteria:**
- Desktop and tablet use centered modal overlay with backdrop
- Mobile uses full-screen modal or bottom sheet where appropriate
- Close button and escape or swipe dismissal are indicated where supported
- Scrollable content area exists when content exceeds viewport
- Focus behavior is defined for open and close actions

### L2-7.4: Accessible Interaction & Keyboard Support
The system shall provide accessible interactions across navigation, forms, dialogs, tables, and notifications.
**Acceptance Criteria:**
- Visible focus indicators are present for keyboard users
- Logical tab order is maintained
- Dialogs trap focus and restore focus on close
- Error messages are associated with the correct form inputs
- Interactive controls have accessible names and sufficient color contrast

### L2-7.5: Loading, Empty, Error, and Success States
Every screen and dialog shall define core UX states.
**Acceptance Criteria:**
- Each major screen has loading, empty, error, and success states
- Retry actions are present where recovery is possible
- Non-blocking progress indicators are accessible
- State layouts are designed for mobile, tablet, and desktop breakpoints

## L1-8: Application Security & Abuse Protection

### L2-8.1: Sensitive Session Storage & Transport
The system shall protect session credentials in transit and at rest in the client.
**Acceptance Criteria:**
- Session credentials are transmitted only over HTTPS
- Sensitive session tokens are not exposed to client-side script-readable persistent storage
- The chosen session mechanism includes CSRF and XSS protections appropriate to the credential transport model
- Session expiry and renewal behavior are documented and testable

### L2-8.2: Abuse Protection & Rate Limiting
The system shall rate-limit and protect sensitive endpoints from automated abuse.
**Acceptance Criteria:**
- Login, sign-up, password-reset, and payment-submission endpoints are throttled per IP and per account where applicable
- Repeated failures trigger backoff or equivalent challenge behavior
- Suspicious activity is logged and can be alerted on
- Discovery-sensitive flows return generic user-facing errors

### L2-8.3: Secure Configuration & Headers
The system shall use secure runtime configuration and browser protection headers.
**Acceptance Criteria:**
- Security headers are defined, including CSP, HSTS, and content-type protection
- Secrets are externally managed and rotatable
- CORS policy is explicitly defined with least-privilege rules
- Audit and application logs do not store raw secrets or full sensitive tokens

### L2-8.4: Sensitive Action Audit Trail
The system shall preserve immutable audit records for security-sensitive actions.
**Acceptance Criteria:**
- Authentication events, admin actions, role changes, loan-term changes, payment postings, reversals, and notification-delivery failures are audit logged
- Audit record includes actor, target, action, timestamp, request identifier, and outcome
- Audit trail is searchable by action type, actor, and target

## L1-9: Financial Integrity & Auditability

### L2-9.1: Authoritative Balance Calculation
The system shall calculate balances consistently from authoritative data.
**Acceptance Criteria:**
- The same balance is shown across loan detail, dashboard, schedule, and notifications
- Balances derive from authoritative schedule, transaction, and allocation records
- Rounding and precision rules are defined consistently
- Currency formatting is consistent across the application

### L2-9.2: Terms & Schedule Version History
The system shall preserve historical versions of loan terms and schedules.
**Acceptance Criteria:**
- Previous loan terms and schedule versions are retained
- Each version change records actor, reason, timestamp, and effective date
- Users can view previous versus current values
- Notifications and reports reference the correct effective version

### L2-9.3: Concurrency & Safe Retry Behavior
The system shall prevent duplicate monetary effects and detect conflicting updates.
**Acceptance Criteria:**
- Retried write operations support idempotency keys or equivalent duplicate protection
- Concurrent edits detect conflicts or serialize safely
- Users receive clear conflict errors when stale data is submitted
- A single business action cannot create duplicate monetary impact

### L2-9.4: Payment Reversal & Correction Handling
The system shall support correcting mistaken payment activity without losing history.
**Acceptance Criteria:**
- Authorized users can reverse or correct a payment without deleting the original record
- Corrected balances recalculate automatically
- Original and compensating entries remain visible in history
- Correction events follow notification and audit policy

## L1-10: Accessibility, Client Quality & Performance

### L2-10.1: Client Route Protection & Session Bootstrap
The client application shall protect routes and bootstrap session state safely.
**Acceptance Criteria:**
- Unauthorized routes are hidden or blocked based on role and session state
- Page refresh re-establishes session state without exposing protected data
- Expired session routes to re-authentication gracefully
- Protected screens do not flash unauthorized content during initial load

### L2-10.2: Client Performance Targets
The client shall provide performant behavior on supported devices and standard network conditions.
**Acceptance Criteria:**
- Login and dashboard screens meet defined first-usable-content targets
- Initial bundle budgets are defined for critical routes
- Large tables and feeds use pagination, incremental loading, virtualization, or equivalent strategy
- User actions show immediate feedback or visible pending state within acceptable UX thresholds

### L2-10.3: Error Recovery & Resilient User Flows
The client shall recover gracefully from transient failures and partial outages.
**Acceptance Criteria:**
- Recoverable request failures present retry options
- Duplicate form submission is prevented during in-flight requests
- User-entered form data is preserved where safe during recoverable failures
- Non-blocking sections can fail independently without breaking the entire page

## L1-11: Reliability, Observability & Operational Readiness

### L2-11.1: Background Processing Reliability
The system shall execute asynchronous work reliably in production.
**Acceptance Criteria:**
- Email and notification jobs run on dedicated worker infrastructure rather than a single web instance
- Jobs support retry, backoff, and failure handling
- Duplicate job execution does not create duplicate user-visible effects
- Scheduled tasks are safe for multi-instance deployment

### L2-11.2: Logging, Metrics & Tracing
The system shall expose the telemetry required to operate production services.
**Acceptance Criteria:**
- Application logs are structured and correlated by request identifier
- Metrics include latency, error rate, queue depth, job failures, and key business-operation outcomes
- Traces or equivalent correlation connect API requests, background jobs, and notification events
- Sensitive data is redacted from logs and traces

### L2-11.3: Health, Backup & Recovery
The system shall support safe monitoring and recovery operations.
**Acceptance Criteria:**
- Liveness and readiness checks are available
- Regular backups are scheduled and restore procedures are tested
- Recovery objectives are defined for data loss and service restoration
- Schema changes have rollback or safe-forward procedures

### L2-11.4: Quality Gates
The system shall enforce automated release quality gates for critical flows.
**Acceptance Criteria:**
- Unit, integration, end-to-end, security, and load tests are defined for critical user journeys
- Database migrations are validated in CI
- Dependency and security scanning are automated
- Releases are blocked on failing critical quality checks

## L1-12: API Contract & Integration Governance

### L2-12.1: Versioned API Contract
The system shall publish a versioned, machine-readable API contract for supported endpoints.
**Acceptance Criteria:**
- Contract includes request and response schemas for all public endpoints
- Request and response examples are provided
- Contract version is kept in sync with implementation changes
- Client integrations can be generated from or validated against the contract

### L2-12.2: Stable Error Semantics
The system shall return consistent, client-safe error structures across APIs.
**Acceptance Criteria:**
- Common error schema is shared across endpoints
- Validation, authentication, authorization, conflict, and rate-limit errors use distinct codes
- Client-visible error messages avoid leaking internal implementation details
- Error responses include a request or correlation identifier

### L2-12.3: Compatibility & Deprecation
The system shall manage API evolution safely for clients.
**Acceptance Criteria:**
- Breaking changes require version change or deprecation window
- Deprecated endpoints or fields are marked before removal
- Write endpoints expected to be retried define idempotency behavior clearly
- Pagination, sorting, and filtering behavior are documented consistently

## L1-13: Bank Accounts & Fund Management

### L2-13.1: Bank Account Overview Screen
The system shall display a bank account overview showing the current balance, recent transactions, and quick actions.
**Acceptance Criteria:**
- Account balance is prominently displayed with currency formatting
- Recent transactions list shows date, description, amount (credit/debit), and running balance
- Quick action buttons for Deposit and Withdraw are visible (Admin only)
- "Set Up Recurring Deposit" action is visible for the account owner and admins
- Desktop: account info card + transaction table side by side; Tablet: stacked; Mobile: stacked single column
- Loading, empty, and error states are defined
- Responsive across all three breakpoints

### L2-13.2: Deposit / Withdraw Dialog
The system shall provide a dialog for admin users to deposit or withdraw funds from a user's bank account.
**Acceptance Criteria:**
- Modal shows account holder name and current balance
- Amount input with currency formatting
- Transaction type selector (Deposit / Withdrawal)
- Optional note/description field
- Validation prevents withdrawal exceeding available balance
- Confirmation preview shows new balance after transaction
- Success, error, and in-flight states are defined
- Desktop/Tablet: centered modal; Mobile: bottom sheet
- Responsive across all three breakpoints

### L2-13.3: Recurring Deposit Setup Dialog
The system shall provide a dialog to set up recurring deposits into a bank account.
**Acceptance Criteria:**
- Modal shows fields for: amount, frequency (Weekly, Bi-weekly, Monthly), start date, and optional end date
- Source description field (e.g., "Allowance", "Paycheck")
- Preview shows next 3 upcoming deposit dates and amounts
- Active recurring deposits are listed on the bank account screen with status (Active, Paused, Completed)
- Edit and cancel actions are available for active recurring deposits
- Desktop/Tablet: centered modal; Mobile: bottom sheet
- Responsive across all three breakpoints

### L2-13.4: Bank Account Transaction History
The system shall display a full transaction history for a bank account with filtering.
**Acceptance Criteria:**
- Table/list shows Date, Description, Type (Deposit/Withdrawal/Recurring), Amount, and Balance After
- Filter by transaction type and date range
- Credit amounts shown in green, debit in red
- Pagination is present for large transaction lists
- Desktop: full table; Tablet: condensed table; Mobile: card list
- Responsive across all three breakpoints

### L2-13.5: Admin Bank Accounts Management List
The system shall provide an admin-only screen listing all users alongside their bank account status, enabling admins to view, search, filter, and take action on any user's bank account.
**Acceptance Criteria:**
- Screen is accessible only to Admin role users
- Table/list shows: User Name, Email, Account Status (Active, Frozen, Closed, No Account), Balance, Last Transaction Date, and Actions
- "No Account" status is visually distinct and shows a "Create Account" action button
- Search bar filters by user name or email
- Filter by account status (All, Active, Frozen, Closed, No Account)
- Sortable columns on desktop (Name, Balance, Status, Last Transaction)
- Clicking a user row or "View" action navigates to the Admin User Account Detail
- Desktop: full data table with all columns; Tablet: condensed table with key columns; Mobile: card-based list
- Loading, empty, and error states are defined
- Responsive across all three breakpoints (375px, 768px, 1280px+)

### L2-13.6: Create Bank Account for User Dialog
The system shall provide a dialog for admins to create a bank account for a legacy user who does not yet have one.
**Acceptance Criteria:**
- Modal shows the target user's name and email for confirmation
- Currency field displays default currency (USD) with option to confirm or change if supported
- Optional initial deposit amount field with currency formatting
- Optional note/reason field for audit trail
- Confirmation summary showing: user identity, currency, and initial deposit (if any)
- "Create Account" primary action and "Cancel" secondary action
- Success state shows the newly created account details with balance
- Error state handles edge cases (user already has account, invalid data)
- Desktop/Tablet: centered modal; Mobile: full-screen or bottom sheet
- Responsive across all three breakpoints

### L2-13.7: Admin User Account Detail View
The system shall provide an admin-specific view of any user's bank account with full management capabilities including deposit, withdraw, view transactions, manage recurring deposits, and change account status.
**Acceptance Criteria:**
- Header shows user identity (name, email, avatar placeholder) and account status badge
- Account summary section shows: Current Balance, Total Deposits, Total Withdrawals, Account Created date
- Quick action buttons: Deposit, Withdraw, Manage Status (freeze/close/reactivate)
- Transaction history table with filtering by type and date range
- Recurring deposits section showing active, paused, and cancelled recurring deposits
- Back navigation to the Admin Bank Accounts list
- Desktop: two-column layout (summary + transactions side by side); Tablet: stacked sections; Mobile: stacked single column
- Loading, empty, and error states are defined for each section
- Responsive across all three breakpoints

### L2-13.8: Account Status Management Dialog
The system shall provide a dialog for admins to change a bank account's status (freeze, close, or reactivate).
**Acceptance Criteria:**
- Modal shows current account status and available status transitions
- Available transitions: Active → Frozen, Active → Closed, Frozen → Active, Frozen → Closed, Closed → Active
- Reason/note field is required for all status changes (audit trail)
- Warning message explains the impact of the status change (e.g., "Freezing will block all deposits and withdrawals")
- Destructive actions (Close) are visually distinct with confirmation step
- Success state confirms the new status
- Desktop/Tablet: centered modal; Mobile: bottom sheet
- Responsive across all three breakpoints

The system shall display all savings goals for the user with progress indicators.
**Acceptance Criteria:**
- Each goal card shows name, target amount, current amount, progress percentage, and optional deadline
- Progress bar visually indicates percentage toward target
- Goals are ordered by closest deadline or most recent activity
- "Create New Goal" button is prominently placed
- Status indicators: In Progress, Completed, Overdue (past deadline, not met)

- Responsive across all three breakpoints

### L2-14.2: Create / Edit Savings Goal Dialog

## L1-15: Recurring Loans

### L2-15.1: Recurring Loan Setup Screen
The system shall allow creditors to configure a recurring loan that auto-generates new loan instances on a schedule.
**Acceptance Criteria:**
- Form fields: Borrower, description template, principal amount, repayment frequency, installment count, interest rate, recurrence interval (Weekly, Bi-weekly, Monthly), start date, optional end date or max occurrences
- Preview shows the next 3 loan instances that would be generated with their start dates
- "Create Recurring Loan" and "Cancel" buttons
- Desktop: two-column form; Tablet: single column; Mobile: single column
- Responsive across all three breakpoints

### L2-15.2: Recurring Loans Management List
The system shall display all recurring loan configurations with status and actions.
**Acceptance Criteria:**
- Table/list shows: Borrower, Description, Amount, Frequency, Status (Active, Paused, Cancelled), Next Generation Date, Total Generated
- Actions: View Details, Pause/Resume, Cancel
- Status badges with distinct colors (Active=green, Paused=amber, Cancelled=gray)
- Desktop: data table; Tablet: condensed table; Mobile: card list
- "Set Up Recurring Loan" button is prominently placed
- Responsive across all three breakpoints

### L2-15.3: Recurring Loan Detail Screen
The system shall display details of a recurring loan configuration and its generated loan instances.
**Acceptance Criteria:**
- Header shows description template, borrower, recurrence status badge, and actions (Edit, Pause/Resume, Cancel)
- Summary cards: Total Generated, Active Loans, Total Principal Disbursed, Next Generation Date
- Generated loans table lists each instance with: Start Date, Status, Principal, Outstanding Balance
- Clicking a generated loan navigates to the standard Loan Detail screen
- Desktop: summary + table; Tablet/Mobile: stacked
- Responsive across all three breakpoints

### L2-15.4: Pause / Resume / Cancel Recurring Loan Dialog
The system shall provide confirmation dialogs for pausing, resuming, or cancelling a recurring loan.
**Acceptance Criteria:**
- Pause dialog shows warning that no new loans will be generated while paused, with optional reason field
- Resume dialog shows next generation date after resumption
- Cancel dialog warns that cancellation is permanent and no future loans will be generated; existing loans are unaffected
- All dialogs show the recurring loan description and borrower for context
- Destructive cancel action is visually distinct
- Desktop/Tablet: centered modal; Mobile: bottom sheet
- Responsive across all three breakpoints
